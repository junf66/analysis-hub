# Codex への検証依頼プロンプト（コピペ用）

> 下のプロンプトを Codex に貼り、`edge_trades.json`（添付バンドル）を一緒に渡す。
> Codex はリポジトリにも J-Quants API にもアクセスできない前提で組んである。
> **検証に必要なデータ（各エッジのトレード明細）は edge_trades.json に全部入っている**ので、
> Codex はそれだけで (1)統計の再計算 (2)銘柄選定の異常検知 (3)方法論の批判 ができる。
> ※ edge_trades.json は `python -m scripts.edge_candidates.verify_edges_standalone --export edge_trades.json` で再生成可。

## ▼ ここから Codex に貼るプロンプト

あなたは独立した検証者です。日本株のクオンツ「エッジ」（短期トレード戦略）7本の主張値を、
私が用意したデータ `edge_trades.json` だけを使って**ゼロから独立に検算・監査**してください。
私（依頼者）側の計算は信用せず、あなた自身の実装で確かめてください。**忖度せず、誤りは誤りと指摘**してください。

### データ構造（edge_trades.json）
```
{
  "cost": {"long": 0.20, "short": 0.15},   // 往復コスト%（方向別）
  "edges": {
    "⑦": {
      "definition": "...",                  // エッジの定義（日本語）
      "direction": "short" | "long",
      "cost": 0.15,
      "claimed": {"n":211,"ev":0.68,"win":58,"t":3.77,"oos":0.43},  // ← 検証対象の主張値
      "recomputed": {...},                  // 依頼者側の再計算（参考。信用せず自分で出す）
      "rows": [[ret_pct, "date", "code", "name", "market"], ...]    // 全トレード明細
    },
    ... 7エッジ
  }
}
```
`rows` の `ret_pct` は **そのトレードの生リターン%**（コスト・方向の符号反転は未適用）。

### タスク1: 統計の独立再計算（rows から自分で計算し claimed と突合）
各エッジについて、`rows` だけから以下を**自分のコードで**計算し、`claimed` と一致するか判定:
- **n** = トレード数
- **net リターン** = `(short なら -ret else ret) - cost`（cost は方向別。short=0.15, long=0.20）
- **EV** = net の平均%
- **勝率** = net > 0 の割合%
- **t_clust（日付クラスタ頑健 t）**: 同一 date 内のトレードは相関するので素朴 t を補正する。式:
  ```
  μ = mean(net)
  各 date g について残差和 S_g = Σ_{i∈g}(net_i − μ)
  G = ユニーク date 数, N = 全件数
  Var(μ) = (G/(G−1)) · Σ_g S_g² / N²
  t_clust = μ / sqrt(Var(μ))
  ```
- **OOS**: date 昇順で先頭70%=train / 後30%=test。方向を train の net 平均符号で決め、test の net 平均%を出す。
- 許容差: EV/OOS ±0.06%, t ±0.12, 勝率 ±2.5pt, n 完全一致。外れたら「主張値が誤り」。

### タスク2: 銘柄選定の異常検知（rows のメタデータを精査）★ここが本番
過去にこの種の汚染で偽エッジが生まれた。`rows` の name/market/ret を見て次を**自分でチェック**:
1. **ETF/ETN/投資信託の混入**: `name` に「ＥＴＦ」「ＥＴＮ」「上場投信」「ファンド」「指数」等を含む銘柄が
   個別株エッジに混ざっていないか（中国株ETF等が過去に+136%/単日のゴミ値を出した）。
2. **単日であり得ない極端リターン**: 全エッジ保有1日（②のみ数日）。個別株で `|ret| > 30%`（特に1日エッジ）が
   あればデータ破損/非個別株の疑い。件数と銘柄を挙げて。
3. **市場区分の整合**: 各エッジの `definition` が指定する市場（例 ⑩R=スタンダード/グロースのみ、①B=プライム中型、
   ②=J-REIT）と `rows` の `market` 列が矛盾していないか。プライム混入等。
4. **少数銘柄/少数日への集中（非独立）**: 同一 `code` や同一 `date` が n の大半を占めて t を水増ししていないか。
   1社1年=1観測に潰して再計算しても有意か（特に n の小さい ①B(34)・⑥(61)・①A(27)）。
5. **生存バイアスの痕跡**: `name` が「?」（マスタ照合不能）の比率。高ければ廃止銘柄絡みの可能性。

### タスク3: 方法論の批判
各エッジの `definition` を読み、次の観点で穴を指摘:
- ルックアヘッド（エントリー時点で未知の情報を使っていないか。寄りエントリー→当日引け決済は寄り時点で発注可能か）
- 過剰最適化（n に対して条件が細かすぎないか。特に候補①A・⑩R）
- コスト前提の妥当性（short 0.15% は逆日歩・空売り規制を無視。②⑦⑩R は実コストで消える可能性。
  各エッジが「コスト何%で t<2 に落ちるか（損益分岐コスト）」も出してほしい）
- 多重検定（7本＋多数の絞り込みを試した中の生き残り。試行回数補正後も有意と言えるか所見を）

### 既知の落とし穴（あなたの監査のキャリブレーション用。これら以外の新規問題を特に探して）
- ⑦: 当初「希薄化(公募)で下がる」と誤解 → 実は「売出し(dilution=0)」が本体だった。
- ④: magnitude 軸の取り違えで 39% 誤分類していた（来期下方率 vs 当期減益率）。
- ⑩（本バンドルには無い・棄却済）: 規模区分を最新スナップショットで過去に遡及適用し**生存バイアス**で偽陽性だった。
- ⑩R: 小型バケツに中国株ETF/ETNが混入し裾を −68% に汚染していた（除去後 −34.9%）。
- 数値の指標混在（α版/raw版/別フィルタ段階を1行に混ぜる）で過去に4箇所誤記した。

### 出力フォーマット
1. **突合表**: エッジ×[n/EV/勝率/t/OOS] の claimed vs あなたの再計算、各 ✅/⚠。
2. **損益分岐コスト**: 各エッジが t<2 に落ちるコスト%。
3. **異常検知の所見**: タスク2で見つけた具体的な銘柄・件数（無ければ「クリーン」と明記）。
4. **方法論の懸念**: タスク3の指摘。
5. **総合判定**: どのエッジが信用でき、どれに疑義があるか。歯に衣着せず。
